OOP

A useful, albeit hard to understand concept.

OOP, or Object Oriented Programming, is a style of programming that, like the name implies, involves treating different parts of the program like “objects.” It is different from procedural programming, where the code is basically followed from top to bottom like the lyrics of a song

The Career Class:

To be used as an example for the tutorial.

public abstract class Career {
    //variables of the class
    private String name;
    private int yearlySalary;
    private String description;
    private int weeklyHours;

    //constructors
    public Career() { //default constructor
        name = "DefaultCareer";
        yearlySalary = 35100;
        description = "";
        weeklyHours = 40;
    } 
    public Career(String name, int yearlySalary, String description, int weeklyHours) { //overloaded constructor
        this.name = name;
        this.yearlySalary = yearlySalary;
        this.description = description;
        this.weeklyHours = weeklyHours;
    }

    //methods
    public abstract void goToWork(); 
    public void returnFromWork() {
        System.out.println("Went home.");
    }
    //accessors
    public String getName() { return name; }
    public int getYearlySalary() { return yearlySalary; }
    public String getDescription() { return description; }
    public int getWeeklyHours() { return weeklyHours; }
    //mutators
    public void setName(String newName) { name = newName; }
    public void setYearlySalary(int newYearlySalary) { yearlySalary = newYearlySalary; }
    public void setDescription(String newDescription) { description = newDescription; }
    public void setWeeklyHours(int newWeeklyHours) { weeklyHours = newWeeklyHours; }
} 

What is an Object?

To understand how OOP works, we must first understand what objects are. So, what are objects?

Objects essentially groups of information and ways to use this information. In more technical terms, objects are comprised of different attributes (variables) and behaviours (methods/functions). They are similar to objects in the real world: take, for example, a water pitcher. The attributes it could have include  waterLevel and liquidContained. These variables would describe the pitcher, since they indicate what amount of the liquid is currently in the pitcher and what said liquid is. Some behaviours it could have include addLiquid() or removeLiquid(). These describe the possible actions the pitcher could do, or the methods or functions of the pitcher. For reference, here's an implementation (or a way of writing) a WaterPitcher class:

public abstract class WaterPitcher {
    private int waterLevel;
    private String liquidContained;
    public WaterPitcher(int w, String l) {
        waterLevel = w;
        liquidContained = l;
    }
    public void addLiquid(int amount) {
        waterLevel += amount;
    }
    public void removeLiquid(int amount) {
        waterLevel -= amount;
    }

    public int getWaterLevel() { return waterLevel; }
    public String getLiquidContained() { return liquidContained; }
} 

Parts of a Class

We've been talking about objects as the basis of Object Oriented Programming. However, there's an important thing to clarify about OOP: objects are merely the instances of classes. Notice that the code that has been shown so far, about the Career class and the WaterPitcher class, all contain the word "class" before the name of that class, and that whenever we refer to these, we always refer to them as classes, not objects. Classes are the blueprints or molds of objects. Instances of objects are created from these molds. In our WaterPitcher example, we can consider the class we wrote a mold for WaterPitcher objects. After writing the class, we still haven't created an object yet; these are created when we type something like:

WaterPitcher w = new WaterPitcher(0, "Water");

There are many components to a class. We will be using the Career class as an example to describe each part.

Class Header

public abstract class Career {

This is the class header. In it, you can specify whether you want the class to be public or private (in this case it's public). You can also specify whether you want it to be a normal class (you don't put anything where abstract is in the example), an abstract class (you put abstract), or an interface (you put interface where abstract is). These terms will be described later, when we discuss superclasses and subclasses. Finally, you must specify the class name, which must follow a naming convention called camel case. Camel case can be achieved when, instead of putting spaces between words, you captialize the first letter of the words. For example, "camel case" would turn into "CamelCase". Note that although everything is named in camel case, there is a difference between naming classes and naming variables and methods with camel case. With classes, the first letter of the first word has to be captalized ("CamelCase"), whereas with the others, the first letter of the first word is not ("camelCase").

Attributes

    //variables of the class 
    private String name; 
    private int yearlySalary; 
    private String description; 
    private int weeklyHours;

Although you can declare all the attributes, or global variables, the class contains anywhere in the class, it's recommended to declare these variables first, before moving to the other parts of the classes, because that makes the code organized and easy to understand. These attributes can be declared like normal variables, except you can define the accessibility of the variable by adding public or private before the type. Public variables can be seen by everyone, including other objects. Private variables, however, only allow for the use of this variable inside the class. It is recommended to put the private keyword in front of the class, since that option offers more security (this can also be called encapsulation).

Constructors

//constructors 
    public Career() { //default constructor 
        name = "DefaultCareer"; 
        yearlySalary = 35100; 
        description = ""; 
        weeklyHours = 40; 
    } 
    public Career(String name, int yearlySalary, String description, int weeklyHours) { //overloaded constructor 
        this.name = name; 
        this.yearlySalary = yearlySalary; 
        this.description = description; 
        this.weeklyHours = weeklyHours; 
    }

Constructor headers look a lot like method headers, with one crucial exception: there is no return type to be specified, and the name must be the same as the name of the class. When an instance of the class is initialized, a constructor is run. A constructor with no parameters is called a default constructor, and constructors can be overloaded (that is, the name of the constructor is the same but the parameter list is not). It is recommended to put contructors immediately after the attributes.

Behaviours

    //methods 
    public abstract void goToWork();

The method header works similarly to the class header, except instead of the keyword "class", we must specify the return type (what type of variable is returned, which can be anything from void, or nothing, to instances of other classes). Another difference is that there can only be abstract methods or normal methods but no interface methods, and that if a method is abstract, there can't be braces (curly brackets) to define the body of a method. Methods can be overloaded as well.

Accessors and Mutators

    //accessors 
    public String getName() { return name; } 
    public int getYearlySalary() { return yearlySalary; } 
    public String getDescription() { return description; } 
    public int getWeeklyHours() { return weeklyHours; } 
    //mutators 
    public void setName(String newName) { name = newName; } 
    public void setYearlySalary(int newYearlySalary) { yearlySalary = newYearlySalary; } 
    public void setDescription(String newDescription) { description = newDescription; } 
    public void setWeeklyHours(int newWeeklyHours) { weeklyHours = newWeeklyHours; }

These get and set methods are what allows other classes to modify and change private global variables. This seems counterintuitve to the purpose of private variables, but these methods still provide more control over the private variables than public classes.

Superclass and Subclasses

An important part of OOP is the interaction between classes. This is done by having classes extend each other (or, if the class being extended is an interface, implementing the interface). This means that the class receiving the extension (the subclass) gets all the variables and methods (not constructors though) contained in the class being extended (the superclass), even the private variables and methods. However, the subclass can only use the public methods and variables in the superclass, not the private variables and methods. You can compare this quirk to recieving a computer and only being allowed to use some of the applications installed on it. The rest of the programs still exist, as they may be important for the computer to function, but they're hidden/unusable to you. These unusable programs are similar to the private variables and methods of a superclass, which can't be called directly but can be called in the body of a public method. 

If a method in the subclass shares the same name and parameter list as a method in the superclass, the subclass method is overriding the superclass method. The effects of this is discussed later

Contrary to normal classes, abstract classes and interfaces cannot have instances created of them, since they may contain abstract methods that don't have bodies, and thus cannot be called properly. They instead rely on the fact that there would be subclasses created with them as the superclasses to be used. They are used primarily to structure and organize other classes. You can consider them to be the "molds" of other classes.

 Let's create a subclass for the Careers class:

The Scientist Class:

The Scientist class is a subclass of Career.

public class Scientist extends Career {
    //variables from Career do not need to be redefined
    String [] qualifications;
    String company;

    //constructors
    public Career() { //default constructor
        super(); //call the superclass constructor
        qualifications = new String [0];
        company = "";
    } 
    public Career(String name, int yearlySalary, String description, int weeklyHours, String [] qualifications,              String company) { //overloaded constructor
        super(name, yearlySalary, description); //call the supercalss constructor
        this.qualifications = qualifications;
        this.company = company;
    }
    //methods (must implement all abstract methods from superclass)
    public void goToWork() {
        System.out.println("Went to work today.");
    }
    public void science() { //Unique method to Scientist
        System.out.println("Did some science today.");
    }
    public void returnFromWork() { //overriding method 
        System.out.println("Safely returned home.");
    }

    //accessors from Career do not need to be redefined
    public String [] getQualifications() { return qualifications; }
    public String getCompany() { return company; }
    
    //mutators from Career do not need to be redefined
    public void setQualifications (String [] newQualifications) { qualifications = newQualtifications; }
    public void setCompany (String setCompany) { company = newCompany; }
} 

Inheritance and Polymorphism

Did you know that subclasses can be casted to superclasses? Just like you can cast an int to a long or a double, you can cast a subclass to a superclass. However, you can't cast a superclass to a subclass because you're going to need information that doesn't exist. Subclasses contain the superclass information as well as its own extra information, meaning that when you cast a superclass to a subclass, you're going to need the add the extra information, which Java doesn't do.

Even stranger is that you can declare a subclass to be a superclass type! Here is an example:

Career w = new Scientist();

This means that, even though the object is actually a scientist, Java recognizes it as a Career object. This creates some interesting situations regarding which functions can or will run. Since the object w is recognized as a Career, methods unique to the Scientist cannot run (so, the science() method cannot be called). However, for overridden functions like returnFromWork(), since w is actually a Scientist, the Scientist version of the function will run. This means that the following statement will print "Safely returned home." (the Scientist version of returnFromWork()) and not "Went home." (the Career version of returnFromWork()):

Career w = new Scientist();
